# PRD 02: Authentication Feature (Register & Login)

## 1. Architectural Overview
This phase implements the core Authentication domain for the `local_service_booking` application. It establishes a secure registration and login pipeline, generates industry-standard JSON Web Tokens (JWT) for session management, and introduces a robust, isolated automated testing framework.

**Design Philosophy:**
We adhere to a Domain-Driven structure. The `users` feature owns the Database Table definitions. The `auth` feature simply imports those models to execute security checks. Cryptographic logic is strictly isolated in `app/core/security.py`.

---

## 2. Core Implementation Components

### A. The Security Layer (`app/core/security.py`)
This file acts as the cryptographic engine for the application.
*   **Password Hashing (`passlib[bcrypt]`):** We utilize the `bcrypt` algorithm to irreversibly hash passwords. The algorithm is configured with `deprecated="auto"` to allow seamless future upgrades to stronger algorithms. Note: `bcrypt` is explicitly pinned to version `3.2.2` in `requirements.txt` to prevent a known crash in `passlib`'s internal wrap-bug detection.
*   **JWT Generation (`PyJWT`):** The `create_access_token` function generates JWTs containing the user's email mapped to the standard `sub` (subject) claim. Tokens are signed using the `SECRET_KEY` and expire based on the `ACCESS_TOKEN_EXPIRE_MINUTES` defined in the `.env` configuration.

### B. The Auth Feature (`app/features/auth/`)
*   **Schemas (`schemas.py`):** 
    *   `LoginRequest`: We explicitly opted for a standard JSON payload over FastAPI's built-in `OAuth2PasswordRequestForm` (which forces `x-www-form-urlencoded` data). This decision significantly improves the developer experience for external frontend frameworks (React, Flutter).
    *   `Token`: Defines the standard output format (`access_token` and `token_type = "bearer"`).
*   **Service Layer (`service.py`):** 
    *   `create_user_service`: Handles the business logic of registration. It actively checks the database for duplicate emails, throwing a `400 Bad Request` if one is found. It then hashes the password and commits the new user.
    *   `authenticate_user`: Handles the business logic of login. It verifies the email exists and passes the password to the security layer for verification.
*   **Router (`router.py`):** 
    *   `POST /auth/register`: Returns a strict `201 Created` status on success. Utilizes the `UserResponse` schema as its `response_model` to automatically strip the hashed password from the output.
    *   `POST /auth/login`: Returns `200 OK`. If authentication fails, it strictly returns a `401 Unauthorized` with the generic message `"Incorrect email or password"` and the required `WWW-Authenticate: Bearer` header.

### C. Application Wiring (`main.py`)
To prevent the application from attempting to connect to the MySQL database during automated testing (which causes crashes during imports), we implemented a **FastAPI Lifespan Event**. 
*   `Base.metadata.create_all(bind=engine)` is executed strictly *during server startup* within the `@asynccontextmanager`. This ensures tables are auto-created when running via Uvicorn/Docker, but safely bypassed when running via Pytest.

## 3. Testing Infrastructure

A comprehensive testing architecture was introduced using `pytest` and `TestClient`.

### A. Test Isolation (`tests/conftest.py`)
*   **SQLite Test Database:** To guarantee that automated tests never mutate or delete production data, a temporary SQLite database (`test.db`) is generated specifically for testing. This file is strictly excluded from version control via `.gitignore`.
*   **Dependency Injection Overrides:** We utilize `app.dependency_overrides[get_db] = override_get_db`. This powerful FastAPI feature intercepts any endpoint requesting a database connection and injects the fake SQLite session instead of the real MySQL session.

### B. Test Suites (`tests/auth/test_auth.py`)
Automated tests simulate an external client to verify system integrity:
1.  **Registration Success:** Verifies a `201` response and ensures passwords are not leaked in the return payload.
2.  **Duplicate Rejection:** Simulates a double-registration attempt and asserts a `400` status code.
3.  **Login Success:** Verifies a `200` response and the presence of a valid `bearer` token.
4.  **Login Failure:** Simulates an incorrect password attempt and asserts a `401` status code.

## 4. API Documentation
The Swagger UI is automatically generated by FastAPI based on the OpenAPI specification and our Pydantic schemas. 
*   Input validation failures (e.g., password length constraints) will automatically return `422 Unprocessable Entity`. This is the intentional and correct behavior in FastAPI, providing frontend clients with a highly structured JSON array (`loc`, `msg`) to identify exactly which form fields failed validation.
